72 /*
73  The 'libzd' compression library provides in-memory compression and
74  decompression functions, including integrity checks of the uncompressed
75  data. This version of the library supports only one compression method
76  (delta_deflation)
77  NOTE: 'libzd' complies with the ZLIB Compressed Data Format Specification
78 
79  Compression can be done in a single step if the buffers are large
80  enough (for example if an input file is mmap'ed), or can be done by
81  repeated calls of the compression function. In the latter case, the
82  application must provide more input and/or consume the output
83  (providing more output space) before each call.
84 
85  The library does not install any signal handler. The decoder checks
86  the consistency of the compressed data, so the library should never
87  crash even in case of corrupted input.
88 */

128 /*
129  The application must update next_in and avail_in when avail_in has
130  dropped to zero. It must update next_out and avail_out when avail_out
131  has dropped to zero. The application must initialize zalloc, zfree and
132  opaque before calling the init function. All other fields are set by the
133  compression library and must not be updated by the application.
134 
135  The opaque value provided by the application will be passed as the first
136  parameter for calls of zalloc and zfree. This can be useful for custom
137  memory management. The compression library attaches no meaning to the
138  opaque value.
139 
140  zalloc must return ZD_NULL if there is not enough memory for the object.
141  If zdelta library is used in a multi-threaded application, zalloc and
142  zfree must be thread safe.
143 
144  The fields total_in and total_out can be used for statistics or
145  progress reports. After compression, total_in holds the total size of
146  the uncompressed data and may be saved for use in the decompressor
147  (particularly if the decompressor wants to decompress everything in
148  a single step).

185 ZEXTERN int ZEXPORT zd_deflate OF((zd_streamp strm, int flush));
186 /*
187  zd_deflate compresses as much data as possible, and stops when the input
188  buffer becomes empty or the output buffer becomes full. It may introduce some
189  output latency (reading input without producing any output) except when
190  forced to flush.
191 
192  The detailed semantics are as follows. zd_deflate performs one or both of
193  the following actions:
194 
195  - Compress more input starting at next_in and update next_in and avail_in
196  accordingly. If not all input can be processed (because there is not
197  enough room in the output buffer), next_in and avail_in are updated and
198  processing will resume at this point for the next call of zd_deflate().
199 
200  - Provide more output starting at next_out and update next_out and avail_out
201  accordingly. This action is forced if the parameter flush is non zero.
202  Forcing flush frequently degrades the compression ratio, so this parameter
203  should be set only when necessary (in interactive applications).
204  Some output may be provided even if flush is not set.
205 
206  Before the call of zd_deflate(), the application should ensure that at least
207  one of the actions is possible, by providing more input and/or consuming
208  more output, and updating avail_in or avail_out accordingly; avail_out
209  should never be zero before the call. The application can consume the
210  compressed output when it wants, for example when the output buffer is full
211  (avail_out == 0), or after each call of zd_deflate(). If deflate returns
212  ZD_OK and with zero avail_out, it must be called again after making room in
213  the output buffer because there might be more output pending.
214 
215  If the parameter flush is set to ZD_SYNC_FLUSH, all pending output is
216  flushed to the output buffer and the output is aligned on a byte boundary, so
217  that the decompressor can get all input data available so far. (In particular
218  avail_in is zero after the call if enough output space has been provided
219  before the call.) Flushing may degrade compression for some compression
220  algorithms and so it should be used only when necessary.
221 
222  If flush is set to ZD_FULL_FLUSH, all output is flushed as with
223  ZD_SYNC_FLUSH, and the compression state is reset so that decompression can
224  restart from this point if previous compressed data has been damaged or if
225  random access is desired. Using ZD_FULL_FLUSH too often can seriously degrade
226  the compression.
227 
228  If zd_deflate returns with avail_out == 0, this function must be called
229  again with the same value of the flush parameter and more output space
230  (updated avail_out), until the flush is complete (deflate returns with
231  non-zero avail_out).
232 
233  If the parameter flush is set to ZD_FINISH, pending input is processed,
234  pending output is flushed and deflate returns with ZD_STREAM_END if there
235  was enough output space; if deflate returns with ZD_OK, this function must be
236  called again with ZD_FINISH and more output space (updated avail_out) but no
237  more input data, until it returns with ZD_STREAM_END or an error. After
238  deflate has returned ZD_STREAM_END, the only possible operations on the
239  stream are zd_deflateReset or zd_deflateEnd.
240 
241  ZD_FINISH can be used immediately after zd_deflateInit if all the
242  compression is to be done in a single step. In this case,
243  a recommended value for avail_out is at least 0.1% larger than avail_in plus
244  12 bytes - on some rare occasions zdelta may expand the data more than this,
245  the application should check the return codes for those cases!
246  If deflate does not return ZD_STREAM_END, then it must be called
247  again as described above.
248 
249  zd_deflate() sets strm->adler to the adler32 checksum of all input read
250  so far (that is, total_in bytes).
251 
252  zd_deflate() returns ZD_OK if some progress has been made (more input
253  processed or more output produced), ZD_STREAM_END if all input has been
254  consumed and all output has been produced (only when flush is set to
255  ZD_FINISH), ZD_STREAM_ERROR if the stream state was inconsistent (for example
256  if next_in or next_out was NULL), ZD_BUF_ERROR if no progress is possible
257  (for example avail_in or avail_out was zero).
258 */
259

297 ZEXTERN int ZEXPORT zd_inflate OF((zd_streamp strm, int flush));
298 /*
299  zd_inflate decompresses as much data as possible, and stops when the input
300  buffer becomes empty or the output buffer becomes full. It may some
301  introduce some output latency (reading input without producing any output)
302  except when forced to flush.
303 
304  The detailed semantics are as follows. zd_inflate performs one or both of the
305  following actions:
306 
307  - Decompress more input starting at next_in and update next_in and avail_in
308  accordingly. If not all input can be processed (because there is not
309  enough room in the output buffer), next_in is updated and processing
310  will resume at this point for the next call of zd_inflate().
311 
312  - Provide more output starting at next_out and update next_out and avail_out
313  accordingly. zd_inflate() provides as much output as possible, until there
314  is no more input data or no more space in the output buffer (see below
315  about the flush parameter).
316 
317  Before the call of zd_inflate(), the application should ensure that at least
318  one of the actions is possible, by providing more input and/or consuming
319  more output, and updating the next_* and avail_* values accordingly.
320  The application can consume the uncompressed output when it wants, for
321  example when the output buffer is full (avail_out == 0), or after each
322  call of zd_inflate(). If inflate returns ZD_OK and with zero avail_out, it
323  must be called again after making room in the output buffer because there
324  might be more output pending.
325 
326  If the parameter flush is set to ZD_SYNC_FLUSH, inflate flushes as much
327  output as possible to the output buffer. The flushing behavior of inflate is
328  not specified for values of the flush parameter other than ZD_SYNC_FLUSH
329  and ZD_FINISH, but the current implementation actually flushes as much output
330  as possible anyway.
331 
332  zd_inflate() should normally be called until it returns ZD_STREAM_END or an
333  error. However if all decompression is to be performed in a single step
334  (a single call of inflate), the parameter flush should be set to
335  ZD_FINISH. In this case all pending input is processed and all pending
336  output is flushed; avail_out must be large enough to hold all the
337  uncompressed data. (The size of the uncompressed data may have been saved
338  by the compressor for this purpose.) The next operation on this stream must
339  be inflateEnd to deallocate the decompression state. The use of ZD_FINISH
340  is never required, but can be used to inform inflate that a faster routine
341  may be used for the single zd_inflate() call.
342 
343  zd_inflate() returns ZD_OK if some progress has been made (more input
344  processed or more output produced), ZD_STREAM_END if the end of the
345  compressed data has been reached and all uncompressed output has been
346  produced, ZD_DATA_ERROR if the input data was corrupted (input stream not
347  conforming to the zlib/libzd format or incorrect adler32 checksum),
348  ZD_STREAM_ERROR if the stream structure was inconsistent (for example if
349  next_in or next_out was NULL), ZD_MEM_ERROR if there was not enough memory,
350  ZD_BUF_ERROR if no progress is possible or if there was not enough room in
351  the output buffer when ZD_FINISH is used.
352 
353  NOTE: there is no check sum for the reference data!
354 */
355
